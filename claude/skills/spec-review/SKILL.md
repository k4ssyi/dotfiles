# 仕様レビュー（Spec Review）

Markdown形式の要件定義書・仕様書を体系的にレビューし、検討不足や見落としを質問リストとして出力するスキル。

## トリガー

「仕様レビュー」「要件チェック」「spec-review」

## ワークフロー

### Step 1: ドキュメントの読み込み

対象の Markdown ファイルのパスを確認し、Read ツールで全文を読み込む。
ファイルが複数ある場合はすべて読み込み、相互参照できる状態にする。

### Step 2: レビュー観点の読み込み

`references/review-checklist.md` を読み込み、レビュー観点を把握する。

```
Read: <skill-path>/references/review-checklist.md
```

### Step 3: 体系的レビューの実施

チェックリストの3カテゴリを順に適用し、ドキュメントの内容を精査する。

#### 3-1. 機能要件の抜け漏れ

以下を検証する：
- **正常系**: ユースケース網羅性、入出力定義、状態遷移、前提/事後条件
- **異常系**: バリデーション、存在しないリソース、権限不足、外部障害、タイムアウト、競合状態
- **境界値**: 数値範囲、文字列長、配列件数、ページネーション、日時境界

#### 3-2. 非機能要件の検討漏れ

以下を検証する：
- **パフォーマンス**: RPS見積もり、レスポンスタイム目標、スケーラビリティ、キャッシュ
- **セキュリティ**: 認証認可、個人情報、サニタイズ、レートリミット、監査ログ
- **運用**: ログ、モニタリング、デプロイ戦略、ロールバック、マイグレーション、フィーチャーフラグ

#### 3-3. ユーザーストーリー間の矛盾・依存関係

以下を検証する：
- **矛盾**: 仕様の不整合、状態遷移の矛盾、用語の不統一、権限モデルの一貫性
- **依存関係**: 実行順序、共有リソース、外部サービス、他チーム連携、リリース順序

### Step 4: 質問リストの出力

検出した問題を **重要度別** に分類し、質問リストとして出力する。

#### 分類基準

| 重要度 | 基準 | 例 |
|--------|------|----|
| Critical | 仕様の矛盾、実装不可能な定義、セキュリティ上の重大な欠落、データ不整合を引き起こす欠落 | 状態遷移の矛盾、エンドポイントの欠落、決済の冪等性未定義 |
| Important | 実装着手時に判断が必要な未定義事項。放置すると手戻りが発生するが、仕様の補足で対応可能 | エラーレスポンスの未定義、境界値の未指定、パフォーマンス目標の不在 |
| Nice to Have | 品質・運用性を高める追加検討事項。開発初期には後回しにしても致命的ではない | 監査ログ、フィーチャーフラグ、段階的ロールアウト |

#### 出力フォーマット

```markdown
## 仕様レビュー結果

### Critical（仕様の矛盾・重大な欠落）

- Q1: [質問文]
  - 該当箇所: [ドキュメント内の参照箇所]
  - 背景: [なぜこの質問が重要か]

### Important（検討が必要な未定義事項）

- Q2: [質問文]
  - 該当箇所: [ドキュメント内の参照箇所]
  - 背景: [なぜこの質問が重要か]

### Nice to Have（あると望ましい追加検討事項）

- Q3: [質問文]
  - 該当箇所: [ドキュメント内の参照箇所]
  - 背景: [なぜこの質問が重要か]
```

## レビュー時の注意事項

- 仕様として **明示的に記載されていない** 事項に注目する。書かれていることの正しさよりも、書かれていないことの発見が主目的
- 質問は具体的に。「セキュリティは大丈夫ですか？」ではなく「認証トークンの有効期限が未定義ですが、セッション管理方式は何を想定していますか？」のように問う
- 質問の総数は 10〜20 件を目安とし、ノイズとなる些末な指摘は省く

### 既存実装との整合性確認

プロジェクトのコードベースが参照可能な場合、以下の範囲で既存実装との整合性を確認する。

**探索範囲（優先順）:**
1. 仕様書のキーワードで Grep し、関連ファイルの存在を把握する。キーワードには仕様書の直接的な用語（機能名、エンティティ名、エンドポイントパス）に加え、機能から推測される既存ドメイン概念（例: 「いいね無制限」→ `canSendLike`, `likeLimit`）も含める
2. 該当する domain 層（エンティティ、値オブジェクト、ドメインサービス、リポジトリインターフェース）を読み、型・状態遷移・ビジネスルールの整合性を確認する
3. use-case 層の主要なユースケースを読み、副作用の範囲を把握する

**探索しない範囲:**
- infra 層の具体的なクエリ実装やモデル定義の詳細
- コントローラー・ルート層の HTTP 詳細
- テストコード

**「直接関連」の判断基準:**
- 仕様書の機能が**依存する**既存コンポーネント（例: ブロック機能が呼び出す `MatchLockClient`）→ 探索対象
- 仕様書の機能を**参照する**既存コンポーネント（例: ブロック状態をチェックする `CanFlickDomainService`）→ 仕様書のビジネスルールが影響範囲に言及していれば探索対象、言及がなければ質問で指摘するに留める
- 仕様書の機能と**無関係な**コンポーネント → 探索しない

**打ち切り基準:** 上記の「直接関連」に該当するファイルの確認が完了した時点で探索を終了する。
